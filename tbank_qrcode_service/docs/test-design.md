# Test Design — tbank_qrcode_service

Применение техник тест-дизайна для микросервиса платежей T-Bank QR.

---

## 1. Boundary Value Analysis (BVA)

**Параметр:** `amount` (сумма платежа)

Граничные значения для поля amount:

| Тест | Значение | Ожидаемый результат |
|------|----------|---------------------|
| Минимальная граница - 1 | -1 | 400 Bad Request |
| Минимальная граница | 0 | 400 Bad Request (сумма должна быть > 0) |
| Минимальная граница + 1 | 0.01 | 200 OK (валидная минимальная сумма) |
| Номинальное значение | 5000.00 | 200 OK |
| Максимальная граница - 1 | 9999999.98 | 200 OK |
| Максимальная граница | 9999999.99 | 200 OK (макс. сумма по API T-Bank) |
| Максимальная граница + 1 | 10000000.00 | 400 или 200 с ошибкой от T-Bank |
| Отрицательная | -500 | 400 Bad Request |
| Нечисловое значение | "abc" | 400 Bad Request |
| Пустое значение | "" | 400 Bad Request |

**Параметр:** `email`

| Тест | Значение | Ожидаемый результат |
|------|----------|---------------------|
| Пустой | "" | 400 Bad Request |
| Без домена | "user@" | 400 Bad Request |
| Без @ | "userdomain.com" | 400 Bad Request |
| Валидный минимальный | "a@b.c" | 200 OK |
| Валидный стандартный | "user@test.com" | 200 OK |
| Максимальная длина (64+255) | "a...@b..." | 200 OK или 400 если > лимита |
| Спецсимволы SQL | "test' OR '1'='1" | 400 (не 500!) |
| Спецсимволы XSS | "<script>alert(1)</script>" | 400 (не 500!) |

## 2. Equivalence Partitioning (EP)

### Классы эквивалентности для `amount`:

|Partition|Диапазон|Представитель|Ожидаемый результат|
|---------|--------|-------------|-------------------|
| Невалидные отрицательные | < 0 | -100 | 400 Bad Request |
| Невалидный ноль | = 0 | 0 | 400 Bad Request |
| Валидные малые | 0.01 — 99.99 | 50.00 | 200 OK |
| Валидные средние | 100 — 99999 | 5000.00 | 200 OK |
| Валидные большие | 100000 — 9999999.99 | 500000.00 | 200 OK |
| Невалидные превышение | > 9999999.99 | 99999999 | 400 / ошибка T-Bank |
| Невалидные тип | не число | "abc" | 400 Bad Request |

### Классы эквивалентности для `userId`:

|Partition|Описание|Представитель|Ожидаемый результат|
|---------|--------|-------------|-------------------|
| Пустой | Отсутствует | (не передан) | 400 Bad Request |
| Валидный короткий | 1 символ | "u" | 200 OK |
| Валидный стандартный | email/uuid | "user-123" | 200 OK |
| Спецсимволы | SQL/XSS | "' OR 1=1" | 400 (не 500!) |

## 3. Decision Table

**Бизнес-правило:** Инициация платежа

| Условие | R1 | R2 | R3 | R4 | R5 | R6 |
|---------|----|----|----|----|----|-----|
| userId передан? | Д | Д | Д | Д | Н | Н |
| amount > 0? | Д | Д | Д | Н | — | — |
| email валидный? | Д | Д | Н | — | — | — |
| amount <= макс? | Д | Н | — | — | — | — |
| **Действие** | **Init OK** | **Ошибка T-Bank** | **400** | **400** | **400** | **400** |

**Тест-кейсы из таблицы:**

| Тест | userId | amount | email | Ожидаемый результат |
|------|--------|--------|-------|---------------------|
| TC-DT-01 | "user-1" | 1000 | "a@b.com" | 200, orderId не пуст |
| TC-DT-02 | "user-2" | 99999999 | "a@b.com" | 200 с error или 400 |
| TC-DT-03 | "user-3" | 1000 | "invalid" | 400 Bad Request |
| TC-DT-04 | "user-4" | -500 | "a@b.com" | 400 Bad Request |

| TC-DT-06 | (пусто) | -100 | (пусто) | 400 Bad Request |

## 4. State Transition

**Сущность:** Платёж

```
  ┌─────────┐    init     ┌─────────┐   QR scan   ┌──────────┐
  │  NEW    │ ──────────→ │ WAITING │ ──────────→ │ PROCESSING│
  └─────────┘             └─────────┘             └──────────┘
                               │                        │
                          timeout                  success
                               │                        │
                               ▼                        ▼
                          ┌─────────┐            ┌──────────┐
                          │ EXPIRED │            │ CONFIRMED │
                          └─────────┘            └──────────┘
                                                        │
                                                   refund
                                                        │
                                                        ▼
                                                   ┌──────────┐
                                                   │ REFUNDED  │
                                                   └──────────┘
```

**Тест-кейсы переходов:**

| Тест | Из | В | Триггер | Ожидаемый результат |
|------|----|---|---------|---------------------|
| TC-ST-01 | — | NEW | Init payment | orderId создан, status = NEW |
| TC-ST-02 | NEW | WAITING | QR отсканирован | Статус меняется |
| TC-ST-03 | NEW | EXPIRED | Таймаут (30 мин) | Платёж просрочен |
| TC-ST-04 | WAITING | PROCESSING | Банк обрабатывает | Статус обновляется |
| TC-ST-05 | PROCESSING | CONFIRMED | Оплата успешна | paymentUrl подтверждён |
| TC-ST-06 | CONFIRMED | REFUNDED | Возврат средств | Refund API вызван |
| Невалидный | EXPIRED | CONFIRMED | Попытка оплатить | Ошибка (нельзя оплатить истёкший) |

## Вопросы для собеседования по этому проекту

**Q: Как ты применял BVA на проекте?**
A: Для поля amount — тестировал границы: 0, 0.01, -1, макс. сумма T-Bank, нечисловые значения. Нашёл, что сервер корректно возвращает 400 на отрицательные значения.

**Q: Сколько тест-кейсов получилось из Decision Table?**
A: 6 правил → 6 тест-кейсов. Decision Table помогла не пропустить комбинацию "все параметры отсутствуют".

**Q: Зачем State Transition, если есть unit-тесты?**
A: Unit-тесты проверяют логику в изоляции. State Transition проверяет, что переходы между состояниями работают в интеграции — например, нельзя оплатить истёкший платёж.
